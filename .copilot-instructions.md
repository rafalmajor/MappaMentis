## General

- Generate code exclusively in **C#** targeting **.NET 8** or newer.
- Assume the project is a **WPF desktop application**.
- Produce production-ready code: readable, maintainable, testable, and aligned with best practices.

---

## Architecture

- Use **MVVM** as the only UI architecture.
- Enforce strict separation of concerns:
  - **View (XAML)** – UI declarations only
  - **ViewModel** – presentation logic
  - **Model / Services** – domain logic and IO
- **No business logic in code-behind** (allowed only: `InitializeComponent` and infrastructure wiring).

---

## MVVM Implementation

- Use **CommunityToolkit.Mvvm**:
  - `ObservableObject`
  - `RelayCommand` / `AsyncRelayCommand`
  - `[ObservableProperty]`
- All ViewModels must implement **INotifyPropertyChanged** (never manually if avoidable).
- Commands must be implemented via **ICommand** only (no event handlers in code-behind).

---

## XAML Rules

- Prefer **Bindings** over direct access from code-behind.
- Avoid `x:Name` unless absolutely necessary.
- Use:
  - **DataTemplates** instead of manually instantiating views
  - **Styles** and **ResourceDictionaries** instead of inline styles
- Keep XAML clean and readable (proper indentation, minimal markup).

---

## Async & Performance

- All IO, networking, and long-running operations must use **async/await**.
- Never block the UI thread (`.Result`, `.Wait()` are forbidden).
- Prefer `AsyncRelayCommand` for asynchronous operations.

---

## Dependency Injection

- Assume **Microsoft.Extensions.DependencyInjection** is used.
- ViewModels and services must be injected via constructors.
- No Service Locator pattern and no static access to dependencies.

---

## Error Handling

- Handle exceptions in **services**, not in Views.
- Validation and error messages belong in the **ViewModel**.
- Prefer well-defined business exceptions or explicit result types.

---

## Testing

- All ViewModels and services must be **unit-testable**.
- Use **xUnit** as the default test framework.
- Use **Moq** (or equivalent) for mocking dependencies.
- Do not test Views (XAML); test behavior and logic in ViewModels and services.
- Asynchronous code must be tested using async test methods (`async Task`).
- Avoid static state and hard dependencies that prevent isolation in tests.

---

## Naming Conventions

- Follow official **C# naming guidelines** consistently.
- **Views**: `{Feature}View` (e.g. `MainView`, `SettingsView`)
- **ViewModels**: `{Feature}ViewModel` (e.g. `MainViewModel`)
- **Models / DTOs**: `{Entity}` or `{Entity}Dto`
- **Services**: `{Feature}Service` or `I{Feature}Service`
- **Commands**: Verb-based names (e.g. `SaveCommand`, `LoadDataCommand`)
- **Async methods** must end with `Async`.
- Interfaces must be prefixed with `I`.
- Names should be explicit and intention-revealing; abbreviations are discouraged.

---

## Output Expectations

- When generating UI:
  - always produce **XAML + corresponding ViewModel**
  - include complete examples (namespaces, usings)
- When making architectural decisions — **briefly justify them**.
- Do not generate WinForms, MAUI, or other UI technologies.
